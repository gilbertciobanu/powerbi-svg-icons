## What It Does

Power BI cannot use image files directly in measures. Icons must be embedded as **data URI strings** inside DAX measures, with the measure's Data Category set to **Image URL**. This skill handles the full pipeline:

- **Design** custom SVG icons from scratch in the Fluent 2 style (Regular and Filled variants)
- **Adapt** existing icons (e.g. from the Fluent 2 library) into the icon set
- **Normalise** Fluent 2 icons with complex geometry via PowerPoint (preserves compound paths)
- **Convert** approved SVGs into ready-to-paste DAX measures
- **Version** every icon and measure with timestamps for traceability

> **See it in action:** `docs/sample-icons.pptx` contains example icons, Power BI renders, and an annotated SVG header — useful for a quick visual overview without running the skill.

> **Note on quotes:** All SVG attributes use **single quotes** throughout — this is by design, not convention. Power BI embeds SVGs as DAX strings wrapped in double quotes; mixing quote types breaks the measure. The skill enforces this automatically and handles the conversion when adapting externally sourced icons.

## Four Workflows

### Workflow A — Create Icons
Design new SVG icons from a concept description. Generates Regular (outline) and/or Filled (solid) variants. Output: `icons/icon-[name]-[variant].svg`

### Workflow B — Convert to DAX
Convert approved SVGs to Power BI DAX measure strings. For SVGs not generated by this skill (e.g. adapted or externally downloaded), the skill runs a safety check before conversion — see the validation note below. Output: `icons/icon-[name]-[variant]_dax.txt`

### Workflow C — Adapt Existing Icons
Take an externally sourced SVG (e.g. from the [Fluent 2 icon library](https://github.com/microsoft/fluentui-system-icons)), recolor it to the project palette, clean attributes for DAX compatibility, and save it with standard naming. The skill checks the source file for potentially harmful content before proceeding — see the validation note below. Output: `icons/icon-[name]-[variant].svg`

### Workflow D — Normalise via PowerPoint
For `ic_fluent_*` icons with compound paths (Regular icons with inner cutouts) where Workflow C's geometry snapping would collapse the cutout boundaries. PowerPoint re-exports the SVG at a clean scale without coordinate distortion. Supports **single-icon** and **batch** modes, and a `--size` flag for custom target resolution (default 96 px). **Requires a one-time setup** — see [One-Time Setup](#one-time-setup-workflow-d) below.

**Single icon — your steps:**
1. Tell Claude the Fluent icon filename and the target name — e.g. *"Normalise `ic_fluent_alert_48_regular.svg` as `icon-alert-regular`"*
2. Claude runs Phase 1 (creates `factory/working.pptx` with 1 slide)
3. Open `factory/macro-seed.pptm` in PowerPoint → **Alt+F8** → run **`ExportIconAsSVG`**
4. A folder picker opens — select the `factory/` folder → OK
5. PowerPoint exports `factory/output.svg` and shows a "Done" dialog
6. Tell Claude **"done"** → Claude runs Phase 2 (adapts and saves to `icons/`)

**Batch — your steps:**
1. List the icons conversationally — e.g. *"Normalise notebook, calendar, folder, laptop (regular)"*
2. Claude runs Phase 1 (creates `factory/working.pptx` with one slide per icon)
3. Open `factory/macro-seed.pptm` → **Alt+F8** → run **`ExportBatchSVGs`**
4. Select the `factory/` folder → OK → PowerPoint exports `output_1.svg`, `output_2.svg`, etc.
5. Tell Claude **"done"** → Claude runs Phase 2 on all icons and shows a summary table

**Custom resolution:** Add "at Npx" to set a non-default target size — e.g. *"Normalise map at 72px"*. Default is 96 px. If the target aspect ratio doesn't match the source, Claude will ask you to confirm.

Output: `icons/icon-[name]-[variant].svg`

---

> **Validation — Workflows B and C (externally sourced SVGs)**
>
> Before processing any SVG not generated by this skill, Claude checks for:
> - `<script>` tags
> - `href`, `src`, or `xlink:href` attributes pointing to external URLs (other than `www.w3.org`)
> - `<image>` tags with an external `href`
> - `<use>` elements referencing external symbol libraries (these do not resolve inside Power BI's sandbox)
>
> If any of the above are found, Claude flags them and asks before proceeding.
>
> **⚠ You are responsible for the final check.** Claude may occasionally miss issues in complex or unusual SVG files. Before deploying an externally sourced icon in production, open the `.svg` file in a text editor and verify it contains no external references or `<script>` elements.

## Which Workflow to Use?

```
Does the icon already exist in the Fluent 2 library (ic_fluent_*)?
├── Yes → Is it a Regular variant with inner cutouts (compound path)?
│           ├── Yes, and Workflow C produces distorted geometry → Workflow D (normalise)
│           └── No / simple shape → Workflow C (adapt)
└── No  → Workflow A (create)

After A, C, or D:
    Happy with the icon? → Workflow B (convert to DAX)
```

Workflows A/C/D and B are intentionally separate steps — don't combine them unless you explicitly ask to.

### Workflow C vs D — when does it matter?

Both workflows take a Fluent 2 source SVG and produce a clean `icon-[name]-[variant].svg`. The difference is how they handle geometry:

- **Workflow C (Adapt)** scales coordinates mathematically (e.g. ×2 for 48→96) and snaps every anchor point to 2px increments. Fast and fully automated, but the snapping can distort thin features — particularly the inner cutout boundaries in compound-path Regular icons, where outer and inner contours are only ~5px apart.

- **Workflow D (Normalise)** delegates geometry to PowerPoint's SVG renderer. No manual scaling, no snapping — the path data comes out exactly as PowerPoint calculated it. The trade-off is an extra manual step (running the VBA macro between Phase 1 and Phase 2).

**Rule of thumb:** if the source icon has compound paths with inner cutouts (most Regular variants), use D. If it's a simple filled shape or a stroke-based outline, C is fine.

## One-Time Setup — Workflow D

Workflow D requires `factory/macro-seed.pptm` — a PowerPoint file containing two VBA macros: **`ExportIconAsSVG`** (single-icon) and **`ExportBatchSVGs`** (batch). This is created once and stays in the `factory/` folder permanently.

**Steps (do once):**

1. Open PowerPoint and create a new blank presentation
2. Press **Alt+F11** to open the VBA editor
3. In the Project tree, right-click the presentation → **Insert → Module**
4. Paste **both macros** into the module — the full VBA source is in `references/workflow-d-macros.md`
5. Save as **macro-enabled presentation** (`File → Save As → PowerPoint Macro-Enabled Presentation .pptm`) into the `factory/` folder, name it `macro-seed.pptm`
6. In PowerPoint **Trust Center** (`File → Options → Trust Center → Trust Center Settings`):
   - Under **Trusted Locations**, add the `factory/` folder
   - Under **Macro Settings**, enable **Trust access to the VBA project object model**

| Macro | Use when | Output |
|---|---|---|
| `ExportIconAsSVG` | Single-icon mode | `factory/output.svg` |
| `ExportBatchSVGs` | Batch mode (multiple icons) | `factory/output_1.svg`, `output_2.svg`, ... |

Claude will tell you which macro to run after Phase 1 completes.

> **Why the folder picker?** OneDrive for Business returns cloud SharePoint URLs from PowerPoint's internal path properties rather than the local Windows path. The folder picker bypasses this by asking you directly. Similarly, files opened directly from an OneDrive-synced folder open in Protected View (shapes become inaccessible), so the macro copies `working.pptx` to `%TEMP%` before opening it.

## Fonts & Glyph-to-Path Conversion

Some icons — pill labels, KPI badges, dynamic text overlays — require text rendered inside the icon shape. The skill supports two approaches with different trade-offs:

| Approach | How it works | Best for |
|---|---|---|
| **SVG `<text>` element** | Text written as a native SVG element; the viewer's system supplies the font at render time | Dynamic text (value changes per row); system fonts like Arial |
| **Glyph-to-path conversion** | Claude uses the `fontTools` library to read a `.ttf` file from the `fonts/` folder and convert each character to an SVG `<path>` node | Specific branded or custom typefaces; text whose value is fixed at icon-design time |

### Why glyph-to-path?

Path-converted text has two important properties:

1. **Font independence** — the icon renders identically on every machine regardless of what fonts are installed. There is no dependency on the viewer's system font stack.
2. **Full SVG compatibility** — Power BI's SVG renderer does not support custom web fonts; path data works everywhere without any font-loading.

The trade-off is cost: the conversion pipeline reads font metrics, calculates per-glyph outlines, scales and positions every anchor point, and writes the resulting `d=` path attribute — this is more computation and produces significantly larger SVG files than a `<text>` element. Longer text strings and heavier font weights increase path complexity further.

### When to use which

Use **glyph-to-path** when:
- You need a specific non-system font (Figtree, Quicksand, Outfit, or any font you add)
- The text content is fixed (a label, a unit, a KPI prefix)
- Visual fidelity across machines matters

Use **SVG `<text>`** when:
- The text value will change per row (DAX-dynamic text, like a measure value)
- Arial Bold or another common system font is acceptable
- You want a smaller, simpler SVG

### Adding a font

Drop the `.ttf` file into the `fonts/` folder following the existing structure:

```
fonts/
└── MyFont/
    └── static/
        └── MyFont-Bold.ttf
```

Then ask Claude to use it by name:

```
Create a "pill-revenue" icon with "Total Revenue" text, font MyFont Bold, converted to path.
```

Claude will detect the file automatically and use it for the conversion.

> **⚠ Fonts and licensing:** Only add fonts you are licensed to embed. Embedding a font as path data in a distributed SVG is generally treated as embedding — check your font's license.

## Versioning

Every SVG file carries a full header in its comment block:

```xml
<!--
  Icon Name  : Bookmark
  Style      : Minimalist, modern, flat (Fluent 2 Filled)
  Variant    : Filled
  Created    : 2026-02-27 08:00:00
  Prompt     : Create a "bookmark" icon, both Regular and Filled variants.
  Summary    : single closed path; Q-bezier rounded top corners, sharp V-notch bottom; stroke-width 5
-->
```

| Field | Purpose |
|---|---|
| `Created` | Timestamp of when this version was saved. Used to name archived files and to stamp the DAX measure. |
| `Prompt` | The exact request you typed — verbatim. Makes it easy to rerun or refine a prompt months later without guessing what you originally asked for. |
| `Summary` | One-line record of the key design decisions: geometry method, corner rounding approach, how the Filled variant was derived, any trade-offs. Useful for understanding *why* the icon looks the way it does, and for prompting consistent iterations. |

> **Using the header to improve your prompts over time:** Compare the `Prompt` field with the `Summary` — the gap between the two shows you what the agent had to decide on its own. If the summary mentions a choice you didn't specify (e.g. `rx='8'`, `Q-bezier tips`, `simplified to 2 shapes`), that's a decision you could have made explicitly in your prompt. Over time, reading summaries helps you build a vocabulary of the design decisions the skill makes, so you can front-load them in future requests and get consistent results on the first attempt. See the [Vocabulary & Prompting Guide](vocabulary.md) for the full list of terms and prompt overrides.

Every DAX `.txt` file echoes the `Created` timestamp in a DAX comment:

```dax
Icon Bookmark Filled =
-- icon-bookmark-filled.svg | 2026-02-27 08:00:00
    "data:image/svg+xml;utf8," & ...
```

When you update an icon and regenerate the DAX, the timestamp changes — so you can see at a glance in the Power BI measure editor which version of the icon is embedded.

## Things to Avoid in Prompts

| Avoid | Why | Better alternative |
|---|---|---|
| Vague concepts | "KPI icon" could be a gauge, trophy, arrow, or chart | Add a visual hint: "gauge-style KPI icon" |
| Skipping the variant | Defaults to Regular — you may want both | Say "both variants" if you need Regular + Filled |
| Asking for multiple icons one at a time | Slower; no batch optimization | Provide a list in a single prompt |
| Combining creation and DAX without saying so | Workflows are separate by default | Explicitly say "then convert to DAX" |
| Describing color in words | "dark blue" is ambiguous | Use the hex code: `#1C3879` |
| Requesting complex paths without a metaphor hint | Ambiguous concepts → more back-and-forth | One short visual description saves one iteration |

## File Conflict Behaviour

**SVG files (Workflows A and C):** If a file with the same name already exists, Claude will show the existing file's `Created` timestamp and ask:
- **Overwrite** — replace the existing file
- **Keep both** — the old file is archived with its timestamp appended to the filename (e.g. `icon-bookmark-regular_2026-02-27-100000.svg`), the new file takes the original name
- **Cancel** — abort

**DAX `.txt` files (Workflow B):** Overwritten silently. Claude notes it in the output: *"Updated existing `icon-bookmark-regular_dax.txt`"*

## Tips for Building a Consistent Icon Set

- **Batch creation is faster than one at a time** — provide a full list in a single prompt; Claude completes one icon before moving to the next, so you can still give per-icon feedback.
- **Mix Workflows A and C** — use the Fluent 2 library for standard icons (folder, mail, settings) and Workflow A for Power BI-specific concepts (KPI, measure, filter state) that don't exist in the library.
- **Add approved icons as reference examples** — the skill uses your best icons as style anchors for future generation. After approving a new icon, ask Claude to add it to the reference examples.
- **Convert to DAX only after approval** — avoid generating DAX from icons you might still iterate on. The timestamp versioning only makes sense once an icon is stable.
- **Keep the `icons/` folder clean** — archived versions (from "keep both") will accumulate. Periodically review and delete superseded files.
- **Use the `ic_fluent_*` prefix as a trust signal** — the agent treats any file named `ic_fluent_*` as a clean, pre-approved source and routes it directly to Workflow C (adapt) or D (normalise), skipping the full creation pipeline. If you have an SVG from another source that you've already reviewed and approved — for example, an icon for documentation or a presentation rather than Power BI — you can rename it to `ic_fluent_<name>.svg` to tell the agent to use it as-is without re-creating it from scratch.
- **Use Workflow D with `--size` to create a design system frame** — when you normalise a batch of icons at the same target resolution (e.g. "normalise star, triangle, alert at 33px"), every icon passes through the same PowerPoint rendering pipeline at the same shape size. The result is a family of icons that share identical viewBox dimensions, consistent stroke rendering, and uniform visual weight — regardless of their original source sizes (24×24, 32×32, 48×48). This is effectively a lightweight design system: one target resolution defines the frame, and every icon in the batch conforms to it. Use this when building icon families for a specific context (e.g. table column icons at 33px, card KPI icons at 72px) where visual consistency across the set matters more than matching the source's native resolution.

## Appendix — Full Design Specifications

### Color

| Property | Value |
|---|---|
| Fill color | `#1C3879` (dark navy) |
| Accent / highlight | `#FFFFFF` (white cutout or inner ring) |
| Opacity variants | 100% primary, 35% decorative/secondary |
| Color mode | Single flat fill — no gradients, no shadows |

Theme palette (data colors — reserve for data series, not icons):
`#4C49ED` `#0EA5E9` `#1C3879` `#607D8B` `#BFCAD0` `#4FD18B` `#AFAEFE` `#D14F4F`

### Size & ViewBox

| Property | Value |
|---|---|
| ViewBox | `0 0 96 96` |
| Width / Height | `100%` — never hardcoded px |
| Optical padding | ~8px inset from edge (shapes within 8–88 range) |
| Aspect ratio | Always 1:1 (square) |

### Style

- Minimalist, modern, flat — Fluent 2 "Regular" weight
- Solid shapes only — no gradients, drop shadows, or glows
- Corners rounded according to shape type — see Corner Rounding below
- Regular (outline/stroke) = idle state; Filled (solid) = active/selected state
- **Filled variants must include `stroke='#1C3879' stroke-width='5'`** to match the Regular variant's visual footprint

### Stroke & Weight

| Property | Value |
|---|---|
| Stroke width | `5` structural; `4` decorative |
| Minimum element | Nothing thinner than `4` (breaks at small sizes) |
| Line cap | `round` |
| Line join | `round` |

> **Note — intentional difference from Fluent 2:** The Fluent 2 source library uses compound paths (outer silhouette minus inner cutout) with a wall thickness of ~5px at the 96×96 scale (2.5px per side). This skill uses `stroke-width='5'` (3px per side) instead. The difference is deliberate: a slightly heavier stroke reads better at the small sizes used in Power BI table and card visuals, where icons compete with data labels and titles. Adapted icons (Workflow C) preserve the Fluent 2 compound-path wall as-is; generated icons (Workflow A) use stroke.

### Pixel Grid

- Snap all anchor points to **2px increments** (e.g. `cx='14'`, not `cx='13.3'`)
- Primary visual center at (48, 48)
- Prefer symmetrical compositions

### Corner Rounding

Corner rounding follows different rules depending on whether the shape is rectangular or polygonal, derived from analysis of the Fluent 2 icon library.

#### Rectangles — tiered `rx`

| Tier | `rx` | When to use |
|---|---|---|
| Large | `rx='12'` | Simple bold shapes that are the whole icon — square badge, folder, calendar |
| Medium | `rx='8'` | Complex icons with multiple internal elements — building, laptop, document |
| Small | `rx='4'` | Secondary elements within an icon (inner panels, labels, slots) |
| Micro | `rx='2'` | Tiny decorative details only |

**Default:** use `rx='8'` when in doubt. Prompt override: specify `rx='12'` (or any value) explicitly.

#### Polygons — straight edges, selective vertex rounding

For shapes with more than 4 vertices (hexagons, pentagons, triangles, stars), use `<path>` rather than `<polygon>`, and apply the following principle: **keep edges straight (`L` commands) and place a small `C` curve only at the sharpest or most pointed vertices.** Wide, obtuse angles can stay sharp; only acute tips need softening.

This preserves the shape's geometric identity at small display sizes and avoids the "rounded blob" effect that comes from rounding every vertex uniformly. It matches the approach used in the Fluent 2 source library.

| Vertex type | Treatment |
|---|---|
| Obtuse angle (>90°, e.g. hexagon flat corner) | Small `C` curve, ~2–3px effective radius |
| Right angle (90°) | Small `C` curve, ~2–3px effective radius |
| Acute angle (<90°, e.g. star tip, triangle apex) | Small `C` curve, ~2–4px effective radius — just enough to avoid a pure point |
| Concave valley (e.g. star inner angle) | `L` command — keep sharp |

**Default:** follow the table above. Prompt override: say "softer corners" or "rounded vertices" to apply larger curves uniformly, or "sharp corners" to skip rounding entirely.

Reference hexagon path (pointy-top, circumradius 36, center 48,48, Fluent 2 style — straight edges, subtle vertex curves):

```xml
<path d='M 42,16 C 48,12 48,12 54,16 L 70,26 C 78,30 78,30 78,38 L 78,58 C 78,66 78,66 70,70 L 54,80 C 48,84 48,84 42,80 L 26,70 C 18,66 18,66 18,58 L 18,38 C 18,30 18,30 26,26 Z'/>
```

All coordinates are ×4 of the legacy 24×24 grid. To convert any existing 24×24 path, multiply every coordinate by 4.

### SVG File Structure

- Single `<svg>` element, no nested `<svg>`
- Comment header inside the file with five fields: `Icon Name`, `Style`, `Variant`, `Created`, `Prompt`, `Summary` (see [Versioning](#versioning))
- No embedded fonts, no external references, no JavaScript
- All attributes use **single quotes** (required for DAX embedding)

### DAX Measure Structure

```dax
Icon Bookmark Regular =
-- icon-bookmark-regular.svg | 2026-02-27 10:00:00
    "data:image/svg+xml;utf8," &
    "<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 96 96' width='100%' height='100%' fill='none'>" &
        "<path fill='none' stroke='#1C3879' stroke-width='5' stroke-linecap='round' stroke-linejoin='round' d='...'/>" &
    "</svg>"
```
